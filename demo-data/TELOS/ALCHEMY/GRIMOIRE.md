# Grimoire

> Governing principles of the practitioner-tool partnership.
> How I work with AI assistance: what it is for, what it is not for, and how we maintain integrity together.

---

## What the Tool Is For

- Accelerating execution of decisions I have already made
- Surfacing options and tradeoffs I may not have considered
- Drafting starting points that I then revise with judgment
- Managing repetitive, structured tasks with high accuracy
- Holding context across a session that I might otherwise lose

## What the Tool Is Not For

- Making decisions about what to build and why
- Determining the ethical dimensions of a project
- Evaluating whether a piece of design or code is *good* (I set the bar)
- Replacing the relational work of understanding a client's actual needs
- Generating the initial creative direction for a visual identity project

## Rules of the Partnership

**1. Evidence before assertion.** The tool must not claim a state exists without having checked it. "The file is correct" requires having read the file. "The test passes" requires having run the test.

**2. Surgical fixes only.** When debugging, make the smallest change that resolves the issue. Do not refactor surrounding code as a side effect. Do not delete components to make a problem go away.

**3. Plan before action on complex tasks.** For anything involving multiple files or steps, present the plan before executing. A surprise implementation is not a gift — it is a loss of control.

**4. The decision belongs to me.** The tool provides counsel. I decide. This is not a limitation — it is the structure that makes the partnership trustworthy.

**5. Coagulation before session end.** Any insight, decision, or lesson worth keeping must be written to a file before the session closes. What was not written did not happen.

## On Transparency

The tool's errors are my responsibility to catch. I do not outsource verification. When the tool produces output, I read it. When I cannot read it, I test it. When I cannot test it, I say so explicitly.
